# RIVERS agent guidance

## Project context

RIVERS is an open inquiry into how organizations can turn meaningful work into durable, reviewable, and reusable knowledge for people and AI agents.

The project is building an evidence-preserving knowledge protocol, not a finished organizational operating system, application, database, schema, or AI product. Its aim is to make the reasoning around work—its purpose, action, verification, learning, future iteration, and possible defaults—available for later people and agents to examine and reuse.

The repository is at an early conceptual stage. `RIVERS-CORE-0001` is the historical, non-normative, supersedable conception of the initial RIVERS lifecycle:

1. Requirement
2. Implementation
3. Verification
4. Education
5. Retry
6. Standardization

This lifecycle is not a finalized standard or a mandatory template. Subsequent CORE records may challenge, refine, or supersede it while preserving the evidence for earlier reasoning. In particular, the possible evolution to QUIVERS is an open question for `RIVERS-CORE-0002`, not an accepted conclusion.

## Repository layout

- `RIVERS/` contains protocol records. Each record directory contains its canonical record and companion evidence.
- `AGENTS/` contains role-specific handovers and working material; it is not a substitute for record evidence.
- `APPLICATIONS/` is reserved for implementations only when justified by completed conceptual work.

## Historical integrity

History is evidence. Do not silently rewrite earlier reasoning to reflect later understanding.

A later record must cite, challenge, refine, supersede, or replace an earlier conclusion. Preserve drafts, criticism, responses, decisions, and disagreement when they explain how understanding changed.

## Source hierarchy

For each record:

1. The designated canonical record is the source of its published claim and status.
2. Companion evidence substantiates the canonical record but does not silently override it.
3. Drafts, reviews, and discussion summaries retain their own historical status.
4. Proposals and unresolved questions are not accepted decisions.

Flag ambiguity; do not resolve it by inference.

## Editing CORE records

Each CORE record must address one primary question or cohesive matter.

Do not amend a historical CORE record with later discoveries. Use a later CORE record unless correcting a demonstrated publication or factual error. Keep the record's status, normative posture, scope, and evidence references internally consistent.

## Evidence and links

Keep evidence inside the record package it supports. Do not create a parallel history system.

Use stable filenames and links that resolve to actual files. Before publication, verify every evidence reference, status statement, and claim of review, approval, or authorship.

## Review and completion

Treat criticism as evidence. Verify changes against the record's stated scope and against Git status.

A change is done only when its purpose, evidence, status, links, and historical boundary are clear; relevant verification has been performed; and no unsupported decision or AI consensus has been implied.
